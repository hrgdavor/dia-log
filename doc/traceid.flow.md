# traceid flow example

Aa concrete scenario: A **Client Application** generates a trace and makes a REST call to **Microservice A**. Microservice A then calls **Microservice B** to gather data before responding.

Here is exactly how the headers, `TraceId`s, and `SpanId`s evolve across the network hops using the W3C Trace Context standard.

---

## The Scenario Setup

Before any calls are made, the Client initializes the trace.

* **The `TraceId` is generated once** and remains identical across the entire lifecycle: `4bf92f3577b34da6a3ce929d0e0e4736`
* Each operation generates its own unique `SpanId`.

---

## Hop 1: Client → Microservice A

The client creates its own span (the "Root Span") and prepares to call Microservice A.

### The IDs at this stage:

* **TraceId:** `4bf92f3577b34da6a3ce929d0e0e4736`
* **Client SpanId:** `00f067aa0ba902b7`

### The Request Header sent to Microservice A:

The client injects the `traceparent` header into its HTTP request payload.

```http
GET /api/v1/data HTTP/1.1
Host: microservice-a.com
traceparent: 00-4bf92f3577b34da6a3ce929d0e0e4736-00f067aa0ba902b7-01

```

> **What this tells Microservice A:** *"You are part of trace `4bf92f3577b34da6a3ce929d0e0e4736`. Your parent is my span `00f067aa0ba902b7`. I have enabled sampling (`01`), so you should record this."*

---

## Hop 2: Microservice A → Microservice B

Microservice A receives the header, extracts the data, and starts its own work. Because it needs more data, it prepares a REST call to Microservice B.

### The IDs at this stage:

* **TraceId:** `4bf92f3577b34da6a3ce929d0e0e4736` *(Unchanged)*
* **Microservice A Parent SpanId:** `00f067aa0ba902b7` *(Read from incoming header)*
* **Microservice A New SpanId:** `88a123bc45de67f8` *(Generated by Microservice A for its own work)*

### The Request Header sent to Microservice B:

When Microservice A calls Microservice B, it creates a *new* `traceparent` header. It keeps the TraceId but replaces the SpanId slot with its own active SpanId.

```http
GET /api/v1/subdata HTTP/1.1
Host: microservice-b.com
traceparent: 00-4bf92f3577b34da6a3ce929d0e0e4736-88a123bc45de67f8-01

```

> **What this tells Microservice B:** *"You are part of trace `4bf92f3577b34da6a3ce929d0e0e4736`. Your parent is my span `88a123bc45de67f8`."*

---

## Inside Microservice B (The End of the Chain)

Microservice B receives the request from Microservice A. It extracts the context and tracks its execution.

### The IDs at this final stage:

* **TraceId:** `4bf92f3577b34da6a3ce929d0e0e4736` *(Still unchanged)*
* **Microservice B Parent SpanId:** `88a123bc45de67f8` *(Microservice A's SpanId)*
* **Microservice B New SpanId:** `999fecba87654321` *(Generated by Microservice B for its local execution)*

---

## Summary Table of the Lifecycle

| Location           | Role / Action                      | TraceId     | Current SpanId | Parent SpanId Passed in Header   |
| ------------------ | ---------------------------------- | ----------- | -------------- | -------------------------------- |
| **Client**         | Generates trace, calls A           | `4bf92f...` | `00f067...`    | *None (Root)*                    |
| **Microservice A** | Receives call from Client, calls B | `4bf92f...` | `88a123...`    | `00f067...` (Client's SpanId)    |
| **Microservice B** | Receives call from A, finishes job | `4bf92f...` | `999fec...`    | `88a123...` (Service A's SpanId) |

When Microservice B finishes, it returns its response payload to Microservice A. Microservice A aggregates the data and returns its payload to the Client. Neither of these standard HTTP *responses* typically require a `traceparent` header, as the tracking data was already captured asynchronously by your OpenTelemetry backend during the request lifecycle.
